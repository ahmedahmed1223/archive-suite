# فصل النظام إلى ثلاثة مستودعات حول حزمة نواة مشتركة — التصميم

> **الحالة:** معتمد (عصف ذهني 2026‑05‑31). يَبني على — ويُحدِّث اتجاه —
> `2026-05-30-spa-cloud-separation-production.md`: بدل monorepo بهدفَي بناء، نفصل
> فيزيائيًّا إلى **ثلاثة مستودعات** حول **حزمة نواة مشتركة**.

## الهدف

فصل تطبيق أرشيف الفيديو إلى مستودعين مستقلّين (نسخة SPA دون اتصال، ونسخة سيرفر
للإنتاج)، يتشاركان منطقًا واحدًا عبر **حزمة نواة** معتمدة بإصدار — مصدر حقيقة واحد
بلا انجراف، مع عزل النشر والإصدار والصلاحيات لكل تطبيق.

## القرارات المعتمدة (من العصف الذهني)

1. **مستودعان للتطبيقين** + **حزمة نواة مشتركة** في مستودعها المستقل (ثلاثة مستودعات).
2. **النشر/الاستهلاك:** اعتماد عبر **وسم git** (`github:ahmedahmed1223/archive-core#vX.Y.Z`)
   — بلا سجل npm ولا رموز مصادقة؛ مثبّت بوسم (ليس `latest`).
3. **الأدوات:** مستقرة/موثوقة فقط — Vite، حزمة `pocketbase` الرسمية، صورة PocketBase
   بوسم إصدار محدّد، Tailwind v4. لا اعتماديات تجريبية.
4. **خادم الإنتاج (نسخة السيرفر):** PocketBase (بيانات+مصادقة+تخزين+لحظي لاحقًا).
5. **نطاق محوّل PocketBase أوّلًا:** محوّل + اختبارات (عميل وهمي محقون) + docker، غير
   مربوط بمسار التشغيل الافتراضي حتى مرحلة لاحقة.

## غير-أهداف (YAGNI)

- تعدّد المستأجرين/الفوترة (النموذج أحادي‑المؤسسة كما هو).
- الزمن الحقيقي/محرّك المزامنة الكامل (مرحلة لاحقة).
- ترحيل بيانات IndexedDB→السحابة (مرحلة لاحقة، عبر حِزَم النقل/الـ delta الموجودة).
- طبقة الذكاء الاصطناعي (مرحلة لاحقة؛ منفذ `AiProvider` جاهز كبذرة).

## طوبولوجيا المستودعات

| المستودع | المسؤولية | البناء/النشر |
|----------|-----------|--------------|
| **`archive-core`** | النواة غير‑المرتبطة‑بخلفية: المنافذ (Storage/File/Auth/Sync/Session/Ai)، المتاجر/الشرائح، الميزات الـ11، الصفحات ومكوّنات الواجهة، نماذج العرض، الأدوات النقية (passwordHash، محرّك المزامنة النقي). **حزمة مكتبة** بلا تطبيق قائم. | `vite --lib` (ESM) + وسوم semver |
| **`archive-spa`** | (المستودع الحالي بعد تقليصه) محوّلات محلية `local-indexeddb`/`files-local`/`local-auth`/`local-sync` + `bootstrap.spa` + المدخل + بناء ملف‑واحد. يعتمد `@archive/core`. | `vite` + `vite-plugin-singlefile` → ملف واحد |
| **`archive-server`** | نسخة الإنتاج: محوّلات `cloud-pocketbase` + `bootstrap.cloud` + خادم PocketBase + Docker. يعتمد `@archive/core`. | `vite` متعدد الملفات + PocketBase |

## مبدأ حدود النواة

- ما **لا يسمّي خلفيةً ملموسة** (IndexedDB/PocketBase/الملف‑الواحد/التشغيل) → نواة.
- ما **يلمس خلفيةً أو تشغيلًا** → تطبيق (محوّلات + bootstrap + إعداد بناء + مدخل).
- **المنافذ** هي السطح العام للنواة؛ كل تطبيق يحقن محوّلاته عبر `register*` في
  bootstrap الخاص به. طبقة المنافذ/المحوّلات المنجزة (PR #79/#81) هي ما يجعل القطع نظيفًا.

## التحلّل إلى مشاريع فرعية (لكلٍّ spec→plan→تنفيذ خاص)

### م.ف 1 — نحت حدود النواة (داخل المستودع الحالي، آمن، بلا مستودعات جديدة)
- تعريف **مدخل النواة العام** `src/core/index.js` (barrel) يعيد تصدير: المنافذ + سجلّ
  المزوّدات (get*/register*) + المتاجر + الميزات + الصفحات التي تستهلكها التطبيقات.
- **بذرة seam الإقلاع:** `src/bootstrap/registerLocalProviders.js` يسجّل المحوّلات
  المحلية صراحةً عبر `register*` (قالبٌ يقابله لاحقًا `registerCloudProviders`).
- **بيان الحدود** `docs/.../core-boundary-manifest.md`: القائمة المرجعية لما هو
  نواة مقابل ما هو قشرة تطبيق (يقود الاستخراج في م.ف 2).
- اختبارات verify للـ seam ولتحلّل مدخل النواة. **verify + البناءان أخضران، بلا تغيير سلوك.**

### م.ف 2 — استخراج `archive-core`
- نقل ملفات النواة فعليًّا تحت `packages/core/` (مرحلة staging بـ workspaces محليًّا)،
  وإصلاح المسارات لتمرّ عبر اسم الحزمة، مع بقاء كل شيء أخضر.
- إنشاء مستودع `archive-core` (عبر `gh repo create` أو `git subtree split -P packages/core`
  لحفظ التاريخ) + إعداد بناء المكتبة + وسم `v1.0.0`.

### م.ف 3 — تحويل المستودع الحالي إلى `archive-spa`
- حذف مصدر النواة، اعتماد `@archive/core` بالوسم، إبقاء محوّلات SPA + بناء الملف‑الواحد
  + `bootstrap.spa`. إعادة تسمية المستودع `archive-app`→`archive-spa`. verify أخضر مقابل النواة المنشورة.

### م.ف 4 — إنشاء `archive-server`
- هيكلة تطبيق Vite يعتمد `@archive/core` + محوّلات `cloud-pocketbase` (StorageProvider
  + SessionProvider عبر عميل PB محقون) + `pb_schema.json` (مجموعات + قواعد وصول
  أحادية‑المؤسسة) + `bootstrap.cloud` + `docker-compose.yml` (PB بوسم محدّد + volume دائم) +
  قائمة فحص يدوي. اختبارات المحوّل بعميل PB وهمي. (يطوي عمل "محوّل PocketBase" هنا.)

## الأدوات والاختبار عبر المستودعات

- **النواة:** تنقل مجموعة `scripts/verify-modules*.mjs` الحالية (تبقى خضراء — تختبر النواة).
- **كل تطبيق:** اختبارات محوّلاته + بناؤه. CI لكل مستودع (lint + test + build) على PRs.
- **محوّلات السحابة:** مصانع تحقن العميل (`create…(client)`) → اختبار حتمي بعميل PB وهمي
  في الذاكرة، بلا شبكة. PB الحقيقي يُجرَّب يدويًّا عبر docker-compose.

## معالجة الأخطاء والمخاطر

- **انجراف الإصدارات:** يُمنع ببناء النواة كمصدر وحيد + تثبيت بالوسم؛ الترقية فعل واعٍ
  (رفع الوسم في package.json).
- **خطر إعادة الهيكلة:** م.ف 1 آمن تمامًا (إضافي، بلا نقل جماعي)؛ النقل الفيزيائي مؤجَّل
  لـ م.ف 2 بخطوات `git mv` مجمّعة مع تحقّق بعد كل خطوة.
- **تسرّب اعتماديات ثقيلة:** حزمة `pocketbase` تُستورد فقط في `archive-server`؛ النواة
  لا تعتمد أي خلفية.
- **اتساق المسارات:** أثناء النقل، يبقى verify + البناءان بوّابة لكل خطوة.

## معايير القبول (للبرنامج)

- ثلاثة مستودعات تبني بنجاح؛ التطبيقان يعتمدان النواة بوسم مثبّت.
- نسخة SPA تبقى ملفًا واحدًا يعمل دون اتصال (سلوك غير متغيّر).
- نسخة السيرفر تتصل بـ PocketBase حيّ عبر docker-compose (يدويًّا) وتنفّذ CRUD+مصادقة.
- لا تسرّب لاعتماديات الخلفية إلى النواة أو إلى حزمة الـ SPA.

## أسماء المستودعات (معتمدة مبدئيًّا)

`ahmedahmed1223/archive-server`. تُنشأ المستودعات الجديدة عند م.ف 2 و م.ف 4 عبر `gh repo create`.
