# تصميم الإعداد السحابي ومدير ملفات FileStore

## الهدف

تبسيط الإعداد الأولي لنسخة Cloud بحيث يكتشف المعالج إعداد الخادم من `.env`، يعبئ نوع قاعدة البيانات ورابط الخادم وبيانات دخول المنصة، ويختبرها قبل إنهاء الإعداد. بالتوازي، يصبح FileStore طبقة تشغيل كاملة يمكن اختيار مزودها من المعالج وإدارة محتواها من صفحة مستقلة، مع صندوق تجهيز يسمح برفع المواد الآن وأرشفتها لاحقًا دون إنشاء عناصر أرشيف تلقائية.

ينقسم التنفيذ إلى ثلاث مراحل مترابطة:

1. الإعداد التلقائي للمعالج وإصلاح pgAdmin.
2. توسيع FileStore وإضافة SFTP وWebDAV وعمليات الملفات والمجلدات.
3. صفحة مدير الملفات وصندوق «بانتظار الأرشفة».

## القرارات المعتمدة

- `.env` هو مصدر الحقيقة الوحيد للقيم الافتراضية والأسرار.
- ملف إعداد خادمي يصف الحقول المسموح للمعالج بقراءتها، ولا يكرر قيم الأسرار داخله.
- بناء على قرار المستخدم، يمكن إرسال كلمة مرور دخول المنصة الافتراضية إلى المعالج في جميع البيئات. تعرض مخفية مع تنبيه واضح بوجوب تغييرها بعد أول دخول.
- أسرار مزودي تخزين الملفات لا ترسل إلى المتصفح. يعرض المعالج الجاهزية والحقول الناقصة فقط.
- PostgreSQL يستخدم نفس نطاق التطبيق افتراضيًا، لذلك لا يلزم إدخال URL في نشر Docker المعتاد.
- العرض الافتراضي لمدير الملفات هو القائمة التشغيلية، مع عرض شبكة بصري اختياري وتذكر آخر اختيار.
- كل رفع يدخل افتراضيًا إلى صندوق «بانتظار الأرشفة». يمكن تعطيل السياسة عالميًا أو تجاوزها لكل عملية رفع.
- لا ينشأ عنصر أرشيف فعلي دون إجراء صريح من المستخدم.

## 1. إعداد المعالج من الخادم

### ملف الإعداد

يضاف `archive-server/config/onboarding.config.js`. يحتوي مخططًا ثابتًا يربط إعدادات المعالج بمتغيرات البيئة، مثل:

- الخلفية: `BACKEND`.
- رابط PocketBase: `POCKETBASE_URL`.
- رابط API العام: `APP_BASE_URL` أو نفس النطاق عند غيابه.
- مستخدم المنصة: `ADMIN_USERNAME`.
- كلمة مرور المنصة: `ADMIN_PASSWORD`.
- سر المصادقة: `JWT_AUTH_SECRET` ثم `JWT_SECRET` كخيار توافق.
- مخزن الملفات: `FILE_STORE`.
- حقول الجاهزية المطلوبة لكل مزود.

الملف لا يحتوي كلمات مرور حرفية. دالة خادمية تقرأ المخطط و`process.env` وتنتج استجابة منظمة.

### API الإعداد

يوسع `GET /api/setup/preset-config` ليعيد دائمًا نتيجة مفيدة، لا فقط عند اكتمال جميع القيم:

```json
{
  "backend": "postgres",
  "serverUrl": "",
  "sameOrigin": true,
  "adminUsername": "admin",
  "adminPassword": "configured-default",
  "mustChangePassword": true,
  "authConfigured": true,
  "database": { "configured": true, "reachable": true },
  "fileStore": {
    "active": "disk",
    "providers": []
  }
}
```

لا تعاد سلاسل اتصال قواعد البيانات أو مفاتيح FileStore السرية. تعاد أسماء المتغيرات الناقصة لكل مزود كي يستطيع المعالج شرح المطلوب.

### سلوك الواجهة

- يجلب المعالج الإعداد مرة عند الفتح في وضع startup.
- يعبئ الخادم المفضل والرابط واسم المستخدم وكلمة المرور تلقائيًا.
- عند تغيير PostgreSQL إلى PocketBase أو العكس، يحمل القيم الافتراضية الخاصة بالخيار الجديد من نفس الاستجابة.
- تبقى كلمة المرور مخفية افتراضيًا، ولا تحفظ في `localStorage`. تستخدم لإنشاء الجلسة ثم تزال من حالة المعالج.
- يعرض تنبيه تغيير كلمة المرور حتى يسجل الخادم أن الحساب لم يعد يستخدم القيمة الابتدائية.
- زر «اختبار الاتصال والمتابعة» يفحص الصحة والمصادقة وقاعدة البيانات، ويعرض نتيجة منفصلة لكل جزء.
- فشل endpoint يعيد المعالج إلى نفس النطاق مع حقول يدوية، ولا يمنع تشغيل الوضع المحلي.

## 2. اختيار FileStore في المعالج

تضاف خطوة مستقلة بعد اختيار قاعدة البيانات. تعرض المزودات التالية:

- Disk أو مجلد مركب، ويشمل NFS المركب بواسطة Docker أو النظام.
- S3-compatible.
- Dropbox.
- Azure Blob Storage.
- Google Drive.
- FTP وFTPS.
- SMB/CIFS.
- SFTP عبر SSH.
- WebDAV.

كل بطاقة مزود تعرض واحدة من الحالات: نشط وجاهز، مهيأ وجاهز، ناقص إعداد، أو تعذر الاتصال. اختيار مزود ناقص يعرض أسماء متغيرات `.env` المطلوبة ولا يعرض الأسرار الحالية.

بعد تسجيل الدخول الإداري يستطيع المعالج حفظ المزود المختار في `server-config.json` عبر API الإدارة. إذا اختلف عن المزود النشط، تعرض الواجهة أن إعادة تشغيل الخادم مطلوبة. لا تدعي الواجهة أن التغيير فُعل قبل إعادة التشغيل.

اختبار المزود غير مدمر منطقيًا: ينشئ ملفًا صغيرًا في بادئة خاصة بالاختبارات، يقرأه، يسرده، ثم يحذفه. ينظف الملف في `finally` حتى عند فشل خطوة لاحقة.

## 3. عقد FileStore الموحد

يبقى العقد الحالي متوافقًا:

- `putBlob(key, blob)`
- `getBlob(key)`
- `getUrl(key)`
- `remove(key)`
- `list(prefix)`
- `describe()`

ويضاف عقد قدرات اختياري:

- `listEntries(path, options)` لإرجاع ملفات ومجلدات مع الاسم والمسار والحجم والنوع والتاريخ.
- `stat(key)`.
- `createFolder(path)`.
- `copy(source, destination)`.
- `move(source, destination)`.
- `removeMany(keys)`.
- `capabilities()` لإعلان العمليات الأصلية.

تقدم طبقة خدمة خادمية fallback موحدًا: النسخ عبر القراءة والكتابة، والنقل عبر النسخ ثم الحذف، عندما لا يملك المزود عملية أصلية. المجلدات الافتراضية في S3 وما يشبهه تمثل ببادئات ولا تسرب تفاصيل المزود إلى الواجهة.

### SFTP

- متغيرات البيئة: `SFTP_HOST`, `SFTP_PORT`, `SFTP_USERNAME`, `SFTP_PASSWORD`, `SFTP_PRIVATE_KEY`, `SFTP_PASSPHRASE`, `SFTP_ROOT`.
- يدعم كلمة المرور أو المفتاح الخاص.
- يرفض traversal وNUL ويقيد كل المفاتيح داخل `SFTP_ROOT`.
- يفتح اتصالًا لكل عملية في البداية لضمان العزل، مع إمكانية pooling لاحقًا إذا أثبت القياس الحاجة.

### WebDAV

- متغيرات البيئة: `WEBDAV_URL`, `WEBDAV_USERNAME`, `WEBDAV_PASSWORD`, `WEBDAV_BEARER_TOKEN`, `WEBDAV_ROOT`.
- يدعم Basic Auth أو Bearer.
- يستخدم عمليات WebDAV القياسية للقائمة وإنشاء المجلد والنسخ والنقل والحذف.
- جميع التنزيلات تمر عبر API المنصة للحفاظ على توحيد المصادقة.

## 4. API مدير الملفات

تضاف مسارات إدارية خلف Bearer token وصلاحية المدير أو المالك:

- `GET /api/files/browser?path=&query=&cursor=&limit=`.
- `POST /api/files/folders` لإنشاء مجلد.
- `POST /api/files/actions` لعمليات copy وmove وrename وdelete وdeleteMany.
- تبقى مسارات الرفع والتنزيل الحالية متوافقة.
- `POST /api/files/test-provider` لاختبار إعداد مزود معين دون تفعيله.

الاستجابات تستخدم أخطاء منظمة: `code`, `message`, `provider`, `retryable`. لا تعاد أسرار أو رسائل خام قد تحتوي URL بكلمة مرور.

## 5. صفحة مدير الملفات

تضاف صفحة مستقلة في التنقل باسم «مدير الملفات»، وليست جزءًا من صفحة الأرشيف. الصفحة عملية وكثيفة:

- شريط أوامر: بحث، رفع، مجلد جديد، تحديث، تبديل القائمة والشبكة.
- شجرة مجلدات في الجانب، مع breadcrumbs للمسار الحالي.
- عرض قائمة افتراضي بأعمدة الاسم والنوع والحجم والتعديل وحالة الأرشفة.
- عرض شبكة اختياري بمعاينات للصور والفيديو والصوت وPDF عند توفرها.
- مفتش للعنصر المحدد يعرض المعاينة والبيانات الوصفية والمسار وإجراءات التنزيل والأرشفة.
- تحديد متعدد وعمليات جماعية: تنزيل، نسخ، نقل، حذف، وإضافة لصندوق الأرشفة.
- إجراءات فردية: معاينة، إعادة تسمية، نسخ، نقل، تنزيل، حذف.
- حالات واضحة للتحميل والتقدم والفشل وإعادة المحاولة.
- على الهاتف تختفي شجرة المجلدات داخل drawer ويفتح المفتش في شاشة جانبية كاملة.

البحث خادمي داخل المسار الحالي افتراضيًا، مع خيار البحث في جميع المجلدات. يجب أن يكون pagination/cursor خادميًا حتى لا تحمل الواجهة آلاف الملفات دفعة واحدة.

## 6. صندوق «بانتظار الأرشفة»

يضاف سجل دائم باسم `file_ingest_queue` في StorageProvider، ويدعم نسخة Cloud في SQL والمزود المحلي عبر التخزين المحلي الحالي. كل سجل يحتوي:

- `id`, `fileKey`, `fileStoreKind`.
- `status`: pending أو archiving أو archived أو ignored أو failed.
- `name`, `mimeType`, `size`, `metadata`.
- `addedAt`, `updatedAt`, `archivedAt`.
- `archiveItemId` بعد نجاح الأرشفة.
- `error` مختصر وآمن عند الفشل.

الإعداد `settings.fileManager.autoQueueUploads` يكون `true` افتراضيًا. نافذة الرفع تعرض checkbox يسمح بتجاوز الإعداد للعملية الحالية. النسخ والنقل لا يضيفان سجلًا جديدًا تلقائيًا، بينما الرفع هو الحدث الذي يطبق السياسة.

عند نقل ملف pending يحدث `fileKey` في السجل. حذف ملف pending يطلب تأكيدًا ويزيل سجل الانتظار أو يوسمه ignored حسب اختيار المستخدم. «أرشفة الآن» يفتح نموذج إنشاء المادة مع الملف والمعلومات المكتشفة، ولا ينشئ السجل النهائي حتى يؤكد المستخدم.

## 7. إصلاح pgAdmin

توجد كلمتا مرور مختلفتان ويجب التعامل معهما صراحة:

1. `PGADMIN_PASSWORD` لدخول واجهة pgAdmin.
2. `POSTGRES_PASSWORD` لاتصال pgAdmin بقاعدة Archive Postgres.

صورة pgAdmin تطبق `PGADMIN_DEFAULT_PASSWORD` عند إنشاء قاعدة إعدادها أول مرة فقط؛ تغيير env لا يعدل مستخدمًا محفوظًا في `pgadmin_data`. يضاف init service يستخدم نفس إصدار الصورة المثبت ونفس volume قبل تشغيل pgAdmin:

- ينشئ قاعدة إعداد pgAdmin عند غيابها.
- يضيف المستخدم عند غيابه.
- ينفذ `setup.py update-user <email> --password <password> --admin` عند وجوده.
- يولد passfile داخل volume من `POSTGRES_USER` و`POSTGRES_PASSWORD` بصلاحيات مقيدة.
- يحمل تعريف Archive Postgres في كل تشغيل باستخدام إعداد الاستبدال المدعوم.

تثبت صورة pgAdmin بإصدار صريح، ويستخدم init والخدمة الرئيسية نفس الإصدار لمنع اختلاف مخطط قاعدة الإعداد.

## 8. الأمان

- قرار إرسال كلمة مرور المنصة للمعالج مقصود ومعتمد، ويظهر تحذير دائم بأنها قيمة أولية يجب تغييرها.
- لا ترسل مفاتيح S3 أو Azure أو Dropbox أو Google أو FTP أو SMB أو SFTP أو WebDAV إلى المتصفح.
- جميع عمليات مدير الملفات تتطلب مصادقة وصلاحية مناسبة.
- يمنع traversal في كل مزود وفي طبقة الخدمة المشتركة.
- الحذف عملية مؤكدة، والحذف الجماعي يعرض العدد والمسار قبل التنفيذ.
- سجلات الخادم تحجب كلمات المرور والمفاتيح وسلاسل الاتصال.

## 9. معالجة الأخطاء

- خطأ إعداد المعالج لا يمنع الوضع المحلي أو الإدخال اليدوي.
- خطأ المصادقة يحدد حقول الدخول ولا يحفظ backend غير صالح.
- خطأ FileStore يبين المزود والعملية وإمكانية إعادة المحاولة.
- العمليات الجماعية تعيد نتيجة لكل عنصر، فلا تضيع نجاحات جزئية.
- النقل fallback لا يحذف المصدر إلا بعد التحقق من كتابة الوجهة.
- الرفع المتقطع يبقى خارج سجل الأرشفة حتى نجاح التخزين.

## 10. الاختبارات والتحقق

### اختبارات منطقية

- resolve إعداد المعالج من `.env` مع `JWT_AUTH_SECRET` و`ADMIN_USERNAME`.
- hydration المعالج وتغيير backend دون بقاء قيم backend السابق.
- جاهزية جميع مزودي FileStore وأسماء المتغيرات الناقصة.
- تحقق config وحفظ المزود مع restart-required.
- عقد SFTP وWebDAV: put/get/list/stat/folder/copy/move/remove.
- fallback النسخ والنقل وسلامة عدم حذف المصدر عند الفشل.
- سياسة auto queue وتحديث المسار بعد النقل.

### اختبارات واجهة

- التعبئة التلقائية، إظهار وإخفاء كلمة المرور، والتنبيه.
- اختبار الاتصال وإنهاء المعالج.
- تبديل القائمة والشبكة مع حفظ التفضيل.
- البحث والتنقل بالمجلدات والتحديد المتعدد.
- الرفع مع تشغيل وتعطيل الإضافة التلقائية للصندوق.
- فتح نموذج الأرشفة من ملف pending.
- RTL وfocus وlabels وعدم تداخل النصوص على الهاتف.

### اختبارات تكامل حية

- pgAdmin على volume جديد وvolume يحتفظ بكلمة مرور قديمة.
- تسجيل الدخول إلى pgAdmin بالقيمة الحالية ثم الاتصال بـArchive Postgres دون خلط كلمتي المرور.
- CRUD ورفع وتنزيل عبر Disk وFTP، واختبار SFTP/WebDAV بحاويات محلية مخصصة.
- بناء Cloud ومسار `/api` عبر نفس النطاق.

## 11. التوافق والترحيل

- العقود الحالية للرفع والتنزيل وFileStore تبقى صالحة.
- `FILE_STORE` الحالي يظل المصدر عند عدم وجود config محفوظ.
- المشاريع والملفات الموجودة لا تحتاج ترحيلًا.
- `autoQueueUploads` يطبق على الرفع الجديد فقط.
- صفحة الأرشيف لا تنشئ عناصر تلقائيًا من الملفات القديمة.

## 12. معيار الإنجاز

يعد العمل مكتملًا عندما يستطيع مستخدم نسخة Cloud فتح المعالج فيجد الخادم وبيانات الدخول ومخزن الملفات معبأة، يختبر الاتصال وينهي الإعداد، ثم يفتح مدير الملفات ويرفع ملفًا وينقله أو ينسخه ويعاينه وينزله وينشئ مجلدًا، ويرى الملف في صندوق الانتظار ويؤرشفه لاحقًا بإجراء صريح. كما يجب أن تعمل كلمة مرور pgAdmin الحالية مع volume جديد وقديم، وأن يتصل pgAdmin بقاعدة PostgreSQL المضمنة باستخدام كلمة مرور قاعدة البيانات الصحيحة.
