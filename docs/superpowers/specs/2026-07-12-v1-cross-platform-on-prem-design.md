# تصميم نشر V1 متعدد الأنظمة داخل المؤسسة

**الحالة:** معتمد من المستخدم بتاريخ 2026-07-12  
**النطاق:** Windows وLinux، باستخدام Docker أو التثبيت Native داخل بيئة On‑Premises

## القرار

تحتوي حزمة V1 على جميع قدرات Archive Suite المقبولة للإصدار، ويختار المستخدم طريقة النشر والقدرات المفعلة من أدوات الإعداد. بيئة النشر الأساسية هي On‑Premises داخل شبكة المؤسسة، بينما Docker وNative طريقتا تشغيل وليستا بديلين لمفهوم On‑Premises.

Docker Compose هو المسار الافتراضي والموصى به لأنه يوفر بيئة متسقة وقابلة للترقية والاستعادة على Windows وLinux. يبقى Native خياراً متاحاً للمؤسسات التي لا تسمح بـDocker، مع فحص متطلبات أكثر صرامة ومصفوفة توافق معلنة.

## فجوة الحالة الحالية

هذا تصميم هدف V1 وليس وصفاً للحالة المنفذة الآن: المسار القانوني الحالي يعتمد Compose بصورة جزئية، بينما لا توجد حزمة Native تشغيلية مكتملة. كما توجد إشارات قديمة إلى PocketBase و`deploy-legacy` وملفات Compose غير قانونية في بعض أدوات الإعداد والوثائق. لا تعلن Native أو أي خيار cross-platform كدعم رسمي حتى تُغلق مهام التوحيد والقبول الواردة في خطة V1.

## الأنظمة ودرجات الدعم

| البيئة | طريقة التشغيل | مستوى V1 |
|---|---|---|
| Windows 10/11 | Docker Desktop + WSL2 | مدعوم رسمياً |
| Windows Server | Docker/WSL2 عند نجاح فحص التوافق، أو Linux VM داخلية | مدعوم ضمن المصفوفة المختبرة |
| Linux Server | Docker Engine + Compose | مدعوم رسمياً وبيئة الخادم المفضلة |
| Windows | Native services | متاح ضمن مصفوفة توافق ودليل مستقل |
| Linux | Native systemd services | متاح ضمن مصفوفة توافق ودليل مستقل |

لا تعلن أي تركيبة مدعومة قبل اجتياز التثبيت والترقية والنسخ والاستعادة والإزالة على مضيف نظيف.

## ملفات التثبيت

### موصى به — الافتراضي

- Docker Compose مع Laravel وNext وPostgreSQL وRedis وworker وHTTPS.
- مؤسسة واحدة لكل تثبيت، حسابات محلية، تخزين دائم ونسخ مجدول.
- القدرات الأساسية كاملة، والخدمات الثقيلة والتكاملات الخارجية مغلقة افتراضياً.

### كامل

- الملف الموصى به، إضافة إلى OCR ومعالجة الوسائط والتفريغ والخدمات الاختيارية المقبولة.
- لا تفعّل خدمة قبل نجاح فحص CPU/RAM/disk/GPU والمنافذ والمفاتيح المطلوبة.
- يعرض المعالج أثر الموارد والمساحة والأمن قبل التأكيد.

### مخصص

- اختيار Docker أو Native، نظام التشغيل، العنوان والمنافذ والشهادة والتخزين وقاعدة البيانات.
- اختيار الخدمات والموارد وسياسة النسخ والاحتفاظ والمراقبة.
- يدعم ملف إعداد declarative لإعادة التثبيت نفسه دون تفاعل.

## أدوات التحكم

- Windows: `Setup-Archive.bat`؛ Linux: `setup.sh`.
- أوامر موحدة: `install`, `doctor`, `start`, `stop`, `restart`, `status`, `health`, `backup`, `restore`, `update`, `rollback`, `uninstall`, `diagnostics`.
- وضع تفاعلي للمستخدم ووضع غير تفاعلي لفرق IT عبر ملف إعداد versioned.
- يفحص `doctor` التوافق والموارد والمنافذ والشهادات والمساحة قبل أي تعديل.
- لا ينفذ `update` migration أو seed تلقائياً؛ يستخدم preflight ونسخة مسبقة وmaintenance وموافقة واضحة.

## بنية الحزمة

- حزمة Online تسحب صوراً ثابتة بالإصدار وdigest.
- حزمة Offline/Air‑gapped تحتوي الصور وcompose والـscripts والقوالب والوثائق وmanifest وchecksums وSBOM.
- profiles مستقلة: `core`, `media`, `ocr`, `ai`, `observability`.
- لا يخزن ملف الإعداد الأسرار الصريحة؛ تنشأ محلياً أو تقرأ من secret store مناسب.
- جميع البيانات في volumes/paths معلنة وقابلة للنسخ والاستعادة والنقل.

## Native

- Linux: systemd units لمكونات الويب والـqueue وReverb والخدمات الاختيارية.
- Windows: خدمات Windows أو service wrapper معتمد، لا نوافذ terminal دائمة.
- PostgreSQL وRedis وruntime versions محددة في compatibility matrix.
- يحفظ المعالج installation manifest لكي تعرف أوامر update/rollback/uninstall ما تملكه دون حذف بيانات المستخدم.

## الأمان والشبكة

- لا خدمة سحابية إلزامية ولا Sentry.
- HTTPS إلزامي عند الوصول الشبكي، مع دعم شهادة المؤسسة.
- لا تكشف PostgreSQL وRedis وworker وReverb للعامة؛ reverse proxy هو المدخل الخارجي.
- يرفض production كلمات المرور الافتراضية و`CHANGE_ME` وdebug وsecure cookies المعطلة.
- الخدمات الاختيارية مغلقة افتراضياً ولا تفتح منافذ حتى تفعيلها.

## الإعداد بعد التثبيت

تعرض لوحة الإدارة حالة الخدمات والموارد والنسخ والإصدار، وتسمح بطلب تغيير profile. التغييرات ذات الأثر الكبير تولد خطة توضح restart/downtime/storage/security impact، ثم تمر عبر Control Center بصلاحيات المشغل.

## الفشل والتراجع

- فشل preflight لا يغير المضيف، وفشل التثبيت ينظف موارد المعالج فقط ويحفظ log منقحاً.
- فشل الترقية يعيد image/package السابق؛ لا تستعاد البيانات إلا من نسخة متحققة وبعد تجربة معزولة.
- `uninstall` يفصل التطبيق عن خيار حذف البيانات؛ الافتراضي الاحتفاظ بالبيانات.

## مصفوفة الاختبار والقبول

كل تركيبة مدعومة تختبر Online وOffline على مضيف نظيف: doctor/install، أول تشغيل، رحلة مادة كاملة، restart للمضيف، backup/restore مع checksums، update/rollback، diagnostics منزوعة الأسرار، ثم uninstall مع الاحتفاظ بالبيانات وإعادة الربط.

يقبل التصميم عندما يكون المسار الموصى به قليل القرارات وآمناً تلقائياً، وتبقى جميع قدرات V1 متاحة دون تشغيل خدمة ثقيلة بلا اختيار وفحص موارد، ويعيد ملف إعداد واحد تثبيتاً مطابقاً، ولا يحتاج التشغيل اليومي للإنترنت، وتظهر طريقة التثبيت والإصدار والـprofiles والـdigests ومسارات البيانات في `status` والـmanifest.

## خارج نطاق V1

- Kubernetes وSaaS متعدد المستأجرين وSSO والخدمات السحابية الإلزامية.
- تركيبات Native خارج compatibility matrix.
- تفعيل AI/GPU تلقائياً دون متطلبات صريحة.
