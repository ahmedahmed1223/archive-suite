# تقرير جاهزية الإصدار الأول لـ Archive Suite

**التاريخ:** 2026-07-12  
**الحالة:** مسودة اعتماد قبل التنفيذ  
**قرار الجاهزية الحالي:** **NO-GO**

## 1. الملخص التنفيذي

وصل المسار القانوني `archive-next` + `archive-laravel` إلى قاعدة منتج متقدمة: واجهة عربية RTL، أرشيف وبحث وإضافة وإدارة، عقد OpenAPI، Docker Compose، وبوابات بناء وأمن أولية. لكنه غير آمن بعد لإطلاق تجاري عام. توجد مانعات مؤكدة قد تسمح بصلاحيات زائدة أو قراءة ملفات من المضيف أو فقد بيانات أثناء اختبار الاستعادة، إضافة إلى تعطل مركز التحكم وعدم وجود حزمة إصدار ثابتة قابلة للتثبيت دون إنترنت.

الاستراتيجية الموصى بها ليست إطلاق `v1.0.0` مباشرة، بل:

1. إغلاق P0 على فرع إصدار نظيف.
2. إصدار `v1.0.0-rc.1` لتجربة داخلية.
3. تجربة ميدانية في 3–5 بيئات شركات منفصلة.
4. إغلاق أعطال P1 ذات أثر الإطلاق.
5. توقيع Product وSecurity وOperations ثم نشر `v1.0.0`.

## 2. تعريف الإصدار الأول

### نموذج النشر المعتمد

- تثبيت On‑Premises محلي أو داخل شبكة الشركة، ومؤسسة واحدة لكل تثبيت.
- دعم Windows وLinux؛ Docker Compose هو المسار الافتراضي والموصى به، وNative خيار متاح عبر أدوات الإعداد ضمن مصفوفة توافق مختبرة.
- ثلاثة ملفات إعداد: موصى به للأساسيات، كامل للخدمات الثقيلة المقبولة، ومخصص لطريقة التشغيل والخدمات والموارد.
- Next.js 16 + React 19 للواجهة، Laravel 13 للخادم، PostgreSQL وRedis، وتشغيل Docker.
- حسابات محلية بأدوار `admin/editor/viewer`؛ لا SSO أو تسجيل دخول خارجي في V1.
- لا Sentry. يعتمد التشغيل على سجلات محلية منظمة، health/readiness، تدوير السجلات، وحزمة تشخيص منزوعة الأسرار.
- جميع قدرات V1 متاحة في الحزمة؛ الذكاء الاصطناعي وOCR والتفريغ الصوتي اختيارية ومغلقة افتراضياً حتى نجاح فحص الموارد والمتطلبات.
- لا ادعاء SaaS متعدد المستأجرين، ولا PWA أو عمل كامل دون اتصال.

### رحلة V1 الأساسية

`التثبيت → إنشاء المدير والمؤسسة → دعوة الفريق → إضافة/رفع مادة → فهرستها → البحث → فتح السجل وتعديله → المشاركة المأمونة → النسخ الاحتياطي والاستعادة → الترقية`

### خارج النطاق

- SSO/Microsoft/Google، SaaS multi-tenant، Kubernetes، Sentry، وتكاملات AI الثقيلة.
- تركيبات Native غير الموجودة في مصفوفة التوافق، والتفعيل التلقائي للخدمات الثقيلة.
- الميزات التجريبية مثل Copilot/plugins/graph/broadcast ما لم تجتز بوابات V1؛ تخفى خلف feature flags ومن التنقل.

## 4. مانعات P0

1. **حساب مدير افتراضي معروف:** أحد compose variants لا يمرر بيانات المدير، فيسقط seeder إلى `test@example.com/password`.
2. **قراءة/كتابة ملفات خارج النطاق:** media jobs تقبل `sourcePath`, `recordId` ومسارات clips دون containment أو ownership/RBAC.
3. **DR drill مدمر:** الاستعادة تعمل على قاعدة حية وتمسح `storage_rows` دون إعادة الحالة الأصلية.
4. **غياب مصفوفة صلاحيات فعلية:** مستخدم viewer يمكنه الوصول إلى عمليات كتابة وإدارة عديدة لأن غالبية API تعتمد auth فقط.
5. **خلط refresh cookie مع جلسة API:** عقد OpenAPI والتنفيذ يسمحان بـ`va_refresh` كبديل bearer عاماً، مع حماية CSRF/throttle غير كافية.
6. **تعطل مركز التحكم:** اختبارات control center أعطت 10 نجاح و13 فشل بسبب `legacyMigrateStatus is not defined`؛ مسار التثبيت والتشغيل غير صالح.
7. **نسخ احتياطي غير كامل:** النسخة الحالية لا تشمل قاعدة PostgreSQL وكل الملفات والكيانات ولا يوجد restore drill معزول موثق.
8. **تشغيل وإزاحة غير آمنين:** `migrate --force` و`db:seed` عند كل start، دون نسخة قبل migration أو maintenance/rollback gate.
9. **لا artifact إصدار قابل للتسليم:** لا offline bundle أو immutable image manifest أو SBOM/checksums/signing.
10. **حالة مصدر غير قابلة لإعادة الإنتاج:** worktree يحتوي أكثر من 1700 تغيير/حذف غير مثبت؛ لا يجوز بناء tag منه.

## 5. P1 قبل الإصدار العام

- جعل onboarding خادمياً: المؤسسة، التخزين، أول عضو، أول مادة، وأول بحث.
- حماية إدارة المستخدمين: منع حذف آخر مدير/النفس، confirmation مع أثر، lifecycle للدعوات وaudit.
- فحص MIME/magic، أسماء UUID، containment، quotas، وفحص برمجيات ضارة للملفات.
- منع كلمات السر في query لروابط المشاركة وتطبيق rate limits فعلياً.
- retention/pruning للجلسات وaudit/jobs وسجلات Docker.
- health/readiness حقيقية لـDB/Redis/queue/storage/worker/Reverb، وتنبيهات محلية دون Sentry.
- إزالة الحدود الصامتة 100/500/1000 أو عرض pagination/aggregation صحيح.
- اختبار 375/768/1280، zoom 200%، keyboard-only، light/dark، وaxe بلا serious/critical.
- توحيد الوثائق على المسار القانوني وحذف أو عزل تعليمات legacy وSentry.
- دليل تثبيت وترقية وتراجع وإزالة ودعم واستعادة، بالعربية.
- اختبار أداء بحجم بيانات V1 متفق عليه، بما فيه ملفات عربية وكبيرة وانقطاع الرفع.

## 6. مخاطر التشغيل وسلسلة التوريد

- خادم PHP المدمج ليس خيار الإنتاج النهائي؛ يعتمد runtime خادماً إنتاجياً موثقاً.
- pin للصور والتبعيات بإصدارات/digests، ومنع استخدام `latest` كمرجع ترقية.
- بناء Next وLaravel واختبار smoke للصورتين قبل النشر.
- إنشاء CycloneDX/SPDX SBOM، فحص dependencies/images/secrets، checksums وتوقيع provenance.
- الصور المنشورة تستخدم version + digest، والـcompose الصادر يستهلكها ولا يبني من المصدر لدى العميل.
- الحزمة offline تشمل الصور والـcompose وقالب البيئة والمثبت والدليل وmanifest/checksums.

## 7. معايير الجودة والقبول

لا يطلق V1 إلا مع دليل آلي محفوظ لكل بند:

- checkout نظيف وtag مطابق للإصدار وChangeLog/release notes وترخيص معتمد.
- `pnpm install --frozen-lockfile`, `pnpm release:verify`, و`pnpm verify:laravel-next:live` كلها خضراء.
- اختبارات RoleMatrix، CSRF/refresh، path containment، upload content، seeder safety، full restore وDR non-destructive خضراء.
- صفر 500 أو console/page errors في الرحلات الحرجة.
- صفر axe serious/critical، ولا horizontal overflow على المقاسات المعتمدة.
- تثبيت offline على Windows نظيف، وتثبيت Docker/Linux نظيف.
- migration rehearsal من آخر إصدار مدعوم، ثم rollback مختبر.
- full backup/restore في بيئة معزولة مع مطابقة counts/checksums وقياس RPO/RTO.
- صور immutable موقعة، SBOM وفحص ثغرات وتراخيص محفوظة مع الإصدار.
- توقيع Product وSecurity وOperations وSupport على Go/No-Go.

## 8. سياسة الإصدار والتراجع

- التسلسل: `v1.0.0-alpha` داخلي → `v1.0.0-rc.1` pilot → RC إضافي عند الحاجة → `v1.0.0`.
- لا migrations مدمرة في RC/GA دون مسار توسعة/نقل/تقليص موثق.
- قبل الترقية: فحص مساحة القرص، نسخة DB+files، maintenance/drain، تشغيل migration مرة واحدة، smoke ثم فتح الخدمة.
- عند الفشل: إيقاف النسخة الجديدة، استعادة image digest السابق، وتطبيق rollback بيانات موثق أو restore للنسخة المعزولة.
- يحتفظ كل إصدار بmanifest يربط نسخة التطبيق وschema وimage digests.

## 9. قرار Go/No-Go

الحالة الحالية **NO-GO**. أول checkpoint يصبح ممكناً فقط بعد إغلاق P0 الأمنية الأربعة الأولى، إصلاح Control Center، واعتماد نسخة احتياطية كاملة واختبارات CI حية من commit نظيف. أي demo داخلي قبل ذلك يستخدم بيانات غير حقيقية وبيئة معزولة فقط.
