# توليد عقود TypeScript وعتبات استخراج الخدمات

## الهدف

إغلاق V1-785 وV1-787 عبر منع انحراف عميل Next عن عقد OpenAPI، وتحديد معايير كمية قابلة للفحص قبل فصل عامل الوسائط أو إضافة outbox. يحافظ التنفيذ على Laravel بوصفه مالك الجدولة والحالة والتدقيق، وعلى واجهة `ArchiveApiClient` الحالية أثناء الانتقال التدريجي.

## القرارات

- يكون `docs/api/archive-contract.openapi.json` مصدر الحقيقة الوحيد لأنواع API المشتركة.
- تستخدم أداة `openapi-typescript` مثبتة الإصدار لتوليد `archive-next/lib/generated/archive-api.ts`؛ لا نكتب مولداً محلياً خاصاً ولا نولد SDK نقل كاملاً.
- يُلتزم الملف المولد في Git حتى يعمل typecheck والبناء دون خطوة توليد ضمنية، وتفشل بوابة drift إذا لم يطابق العقد.
- يبقى عميل النقل اليدوي مستقراً. تبدأ الهجرة بنماذج السجلات والمرفقات وأغلفة الاستجابة، ثم يمكن ترحيل بقية الأنواع على دفعات مستقلة.
- تحفظ عتبات الاستخراج في `docs/architecture/service-extraction-thresholds.v1.json` كعقد machine-readable، ويشرح ملف Markdown المصاحب طريقة القياس والقرار والتراجع.
- لا يؤدي تجاوز مؤشر منفرد أو قياس قصير إلى تغيير معماري؛ يلزم تجاوز العتبات المحددة طوال نافذة القياس، مع دليل benchmark أو production-like قابل لإعادة التشغيل.

## توليد TypeScript

### التدفق

1. يشغّل `pnpm api:generate` الأداة المثبتة على عقد OpenAPI ويكتب الناتج في المسار القانوني المولد.
2. يشغّل `pnpm verify:api-generated` سكربتاً لا يغيّر workspace: يولد إلى مجلد مؤقت، يطبع diff موجزاً عند الاختلاف، ثم يحذف المؤقت في `finally`.
3. تسبق بوابة drift الـtypecheck داخل `pnpm verify` حتى تكون رسالة الفشل مباشرة.
4. يستورد `archive-next/lib/archive-api.ts` aliases من الأنواع المولدة للسجلات والمرفقات وطلبات إنشاء السجل، مع إبقاء الأسماء العامة الحالية مستقرة للمستهلكين.

### حدود التوليد

- لا تُعدل الملفات المولدة يدوياً.
- لا يعتمد runtime في المتصفح على `openapi-typescript`؛ هي devDependency فقط.
- لا تنشئ البوابة اتصالات شبكة أو timestamps أو مسارات مطلقة، ولذلك يكون الناتج deterministic بين Windows وCI.
- إذا لم يصف OpenAPI شكلاً يستخدمه العميل، يُصلح العقد أولاً بدلاً من إضافة نسخة يدوية جديدة للشكل نفسه.

## عقد عتبات استخراج الخدمات

يتضمن العقد إصدار schema، نافذة قياس لا تقل عن 14 يوماً أو benchmark ثابت مكافئ، وحداً أدنى لحجم العينة، ومجموعتين من القرارات:

### فصل عامل الوسائط

يوصى بفصل العامل فقط عند تحقق اثنين على الأقل من المؤشرات التالية خلال النافذة نفسها:

- P95 لانتظار طابور الوسائط يتجاوز 120 ثانية.
- معدل الإدخال المستدام يتجاوز 1,000 وظيفة وسائط يومياً.
- انشغال موارد CPU أو GPU يتجاوز 75% لمدة أربع ساعات يومياً في ثلاثة أيام أو أكثر ضمن النافذة.
- معدل فشل الوظائف القابل لإعادة المحاولة يتجاوز 2% بعد استبعاد أخطاء الإدخال الدائمة.
- تتطلب سعة CPU وGPU دورات نشر أو scaling مستقلة مثبتة بالدليل.

يبقى Laravel مالك `media_jobs` والجدولة والحالة والتدقيق. العامل المستقل يستهلك عقد وظيفة versioned ويعيد نتائج idempotent؛ لا يملك API العامة ولا قاعدة بيانات المنتج مباشرة.

### إضافة transactional outbox

تضاف outbox فقط عند وجود تكامل خارجي واحد على الأقل يحتاج تسليماً موثوقاً، ومع تحقق أحد الآتي:

- فقد أو ازدواج حدث واحد على الأقل في اختبار game-day موثق بسبب الفجوة بين transaction والنشر.
- حجم أحداث مستدام يتجاوز 100,000 حدث يومياً مع مستهلكين متعددين.
- شرط تعاقدي مثبت للتسليم at-least-once والتتبع وإعادة التشغيل.

يُلزم التصميم عندها بمفتاح idempotency، retention، replay محدود، dead-letter، ومقاييس age/attempts/failures. لا يُعتمد event bus افتراضياً لمجرد وجود outbox.

## التحقق والأخطاء

- اختبار Node يثبت أن التوليد الحالي مطابق للعقد، ويفشل عند تغيير نسخة مؤقتة من OpenAPI دون إعادة التوليد.
- اختبار TypeScript يثبت أن aliases العامة المختارة مشتقة من `components["schemas"]` وأن واجهة العميل الحالية ما زالت قابلة للنوع.
- اختبار Node يتحقق من schema العتبات: الإصدار، الوحدات، النطاقات، النوافذ، عدد المؤشرات المطلوب، وعدم تعارض قواعد الفصل/outbox.
- رسائل الفشل تذكر الأمر التصحيحي (`pnpm api:generate`) أو الحقل غير الصحيح، ولا تطبع العقد كاملاً.
- تمر `pnpm verify` الكاملة على Node 25.9.0 قبل إغلاق المهام.

## التوثيق وإغلاق المهام

- يشرح `docs/architecture/service-extraction-thresholds.md` كيفية جمع الأدلة، من يراجع القرار، وكيفية الرجوع إذا لم يحسن الفصل SLOs.
- تزال V1-785 وV1-787 من `TASKS.md` فقط بعد نجاح البوابة، وتؤرشف التفاصيل في `ChangeLog.md`.
- يكون لكل ميزة commit مستقل بعد commit التصميم: واحد لتوليد OpenAPI، وآخر لعقد عتبات الاستخراج.

## خارج النطاق

- توليد SDK HTTP كامل أو تغيير signatures العامة للعميل.
- فصل عامل الوسائط فعلياً، إضافة message broker، أو تطبيق outbox الآن.
- تغيير API العامة أو مخطط `media_jobs`.
- الاعتماد على قياسات إنتاج غير متوفرة لإغلاق V1-787؛ المطلوب هنا عقد القرار وأداة التحقق، لا ادعاء أن العتبات تجاوزت.
