# تصميم مصفوفة القبول التفاعلي للإصدار الأول

**التاريخ:** 2026-07-19  
**الحالة:** معتمد مبدئيًا من صاحب المنتج، بانتظار مراجعة المواصفة المكتوبة  
**النطاق:** المسار القانوني `archive-next` + `archive-laravel`، وتشغيل Docker وWindows وLinux

## الهدف

إنشاء برنامج قبول تفاعلي قابل للتكرار يثبت أن إصدار Masar الأول يعمل كمنظومة كاملة، لا كمجموعة اختبارات وحدات منفصلة. يغطي البرنامج دورة حياة المنصة، رحلات مدير النظام، أعمال الأرشفة، المونتاج، الأحمال الكبيرة، وتعدد المستخدمين. تُنفّذ السيناريوهات حسب الإمكانات المتوفرة محليًا، وتُصعّد إلى أجهزة أو آلات افتراضية نظيفة قبل RC وGA.

## النهج المعتمد

تُقسم الاختبارات إلى ثلاث طبقات:

1. **طبقة يومية محلية:** Docker على جهاز التطوير، وWindows Native عند توفر runtime صالح. تُشغّل السيناريوهات الحرجة والسريعة بعد الدفعات المؤثرة.
2. **طبقة دورية مؤتمتة:** Windows وLinux داخل Hyper-V أو WSL2 أو VM بديل قابل للأتمتة. تُشغّل المصفوفة التشغيلية ورحلات الأدوار والأحمال المتوسطة.
3. **طبقة قبول الإصدار:** أجهزة أو VMs نظيفة، online وoffline، تثبت من artifacts موقعة لا من workspace، وتنفذ مصفوفة RC/GA الكاملة بما فيها التحديث والتراجع والنسخ والاستعادة.

لا يُعتبر Docker بديلًا عن قبول Windows Native أو Linux Native. إذا تعذر hypervisor معين، يختار المشغل بديلًا قابلًا للأتمتة ويسجل provider والإصدار والموارد في دليل التشغيل.

## نموذج السيناريو والدليل

لكل سيناريو معرّف ثابت بالشكل `V1-IA-<المجال>-<الرقم>`، ويحتوي تعريفه على:

- المنصة، نمط التثبيت، profile، موارد الجهاز، وحالة الشبكة.
- المستخدم/الدور والبيانات الاصطناعية المطلوبة.
- المتطلبات المسبقة وخطوات الإعداد والتنفيذ والتنظيف.
- نقاط تحقق آلية ونتيجة متوقعة قابلة للحسم.
- حدود زمنية وأداء عند انطباقها.
- artifacts الدليل: JSON، screenshots، traces، logs، manifest، checksums، ولقطة صحة.
- سياسة إعادة المحاولة وتصنيف الفشل: منتج، منصة، بيانات، بيئة، أو flake.

يُنشأ manifest تشغيل واحد لكل run يربط commit ونسخة التطبيق وdigest الصور والحزم بالسيناريوهات والنتائج. تُنزع الأسرار والمسارات الحساسة من جميع الأدلة قبل حفظها.

## حزم القبول

### 1. المنصة ودورة الحياة

تغطي Docker وWindows Native وLinux Native في أوضاع local/intranet/public TLS وعند توفر الشبكة أو انقطاعها:

- تثبيت أول، first-run، تشغيل، إيقاف، وإعادة تشغيل الجهاز.
- إعادة الاتصال بعد reboot وتعافي الخدمات بالترتيب الصحيح.
- تثبيت offline من bundle موثق دون registry أو تنزيل خفي.
- update إلى إصدار أحدث ثم rollback مع حفظ البيانات والإعدادات.
- uninstall مع خيار الاحتفاظ بالبيانات، ثم reconnect أو إعادة التثبيت.
- فشل منفذ أو شهادة أو DNS أو مساحة قرص أو خدمة تابعة، مع رسالة قابلة للتشخيص وتعافٍ موثق.

### 2. رحلة مدير النظام

تبدأ من جهاز نظيف وتنتهي بنظام صالح لتسليمه للمستخدمين:

- اختيار نمط التشغيل، التخزين، قاعدة البيانات، TLS، البريد، والميزات الاختيارية.
- إنشاء admin وإدارة editor/viewer والدعوات وتغيير الأدوار ومنع فقدان آخر admin.
- مراجعة الصحة والطوابير والموارد والسجلات وسجل التدقيق وحزمة الدعم.
- تشغيل backup، verify، restore preview، restore فعلي، وقياس RPO/RTO.
- صيانة cache والـmigrations والطوابير والاحتفاظ والتنظيف دون فساد البيانات.
- تشخيص DB/cache/queue/worker/Reverb/network/disk/certificate، ثم تطبيق runbook والتحقق من التعافي.
- التحقق من أن الإجراءات الحساسة تعرض preview وتطلب تأكيدًا وتُسجل تدقيقيًا.

### 3. رحلة موظف الأرشفة

تغطي دورة المادة من الوصول إلى الاسترجاع والمشاركة:

- إدخال ملف مفرد، ملفات متعددة، مجلد، drop، استيراد URL، وبيانات وصفية بلا ملف.
- استئناف ملف كبير عند توفر الرفع المجزأ، والتحقق من checksum والمرفقات والمشتقات.
- توصيف عربي، أنواع وحقول وقوالب ووسوم ومفردات وعلاقات وموقع وحقوق.
- ملاحظات شخصية وتعليقات فريق وإشارات وحضور وأقفال وتعارض تحرير.
- بحث عادي ومتقدم ودلالي وداخل التفريغ، autocomplete، facets، بحث محفوظ ومشارك، وروابط زمنية.
- مجموعات ومفضلة ووارد وقراءات وكانبان ومشاريع، مع تصدير CSV والملفات والروابط العامة المقيدة.
- حذف واستعادة من السلة، undo، وسجل تغييرات صحيح لكل عملية.
- تحقق صلاحيات viewer/editor/admin في كل انتقال حرج.

### 4. رحلة المونتاج والتجميع والتصدير

تغطي رحلة إنتاج قابلة للتسليم:

- إنشاء مشروع مونتاج من نتائج البحث أو مجموعة.
- إضافة المقاطع وترتيبها وقصها وإضافة الانتقالات وحفظ المسودة واستعادتها.
- مراجعة زمنية وتعليقات ورسومات annotation وروابط مراجعة بصلاحيات وانتهاء.
- تشغيل thumbnail/transcode/transcription والانتظار والطوابير والإلغاء وإعادة المحاولة.
- تصدير الناتج والصيغ والبيانات المرافقة والتحقق من checksum ومسار التنزيل.
- فشل FFmpeg أو worker أو القرص في منتصف العمل، مع بقاء المشروع قابلًا للاستئناف وعدم إعلان نجاح كاذب.

### 5. الأحمال والبيانات الكبيرة

تستخدم بيانات اصطناعية ببذرة ثابتة ولا تعتمد على بيانات مؤسسة حقيقية:

- 100 ألف سجل عربي، 10 آلاف ملف، وعينة رفع 1GB وفق مهمة benchmark المعتمدة.
- آلاف عمليات الإدخال والتوصيف والبحث والتصدير، مع دفعات متزامنة وطوابير ممتلئة.
- قياس LCP/CLS/INP وP95 للبحث وفتح السجل وبدء الرفع وفق budgets الإصدار.
- مراقبة الذاكرة وCPU والقرص وعمق الطوابير وأقدم job ومعدل الخطأ.
- soak run طويل يكشف التسرب والتراكم، ثم يثبت عودة الموارد والطوابير إلى حالة مستقرة.
- فشل الاختبار عند فقد سجل، تكرار غير مبرر، checksum خاطئ، job عالق، أو تجاوز budget بلا استثناء موثق.

### 6. تعدد المستخدمين والتزامن

تستخدم جلسات مستقلة ومتزامنة لمستخدمين حقيقيين من fixtures معزولة:

- admin/editor/viewer يعملون في الوقت نفسه على سجلات ومشاريع ومجموعات مشتركة وخاصة.
- اختبار الحضور الفوري، lock acquire/refresh/release، conflict، وانتهاء القفل.
- تعديلان متزامنان، نسخة stale، حل التعارض، وعدم ضياع آخر نسخة مقبولة.
- مشاركة بحث أو مجموعة أو مشروع، وتفويض مؤقت ينتهي تلقائيًا عند تنفيذ الميزة.
- وصول الإشعارات للمستخدم الصحيح مرة واحدة، وعمل الروابط العميقة، وعدم تسرب المحتوى.
- عزل الملاحظات الشخصية، تطبيق field ACL والحقوق، وتدقيق actor/outcome دون أسرار.
- خروج مستخدم أو انقطاعه في منتصف عملية، ثم تنظيف الحضور والأقفال واستئناف العمل بأمان.

## المعمارية المقترحة للأتمتة

### منسق السيناريوهات

أداة Node في `scripts/` تقرأ تعريفات السيناريو، تتحقق من القدرات المتوفرة، تنفذ adapters، تجمع الدليل، وتصدر summary موحدًا. لا تنفذ منطق المنتج، بل تنسق الأدوات القائمة مثل `pnpm dev` وDocker وPlaywright وcontrol-center وطلبات الصحة.

### موفرو البيئة

- `docker`: تشغيل Compose أو الحزمة القانونية وانتظار health/readiness.
- `windows-native`: استخدام runtime وخدمات Windows الحالية.
- `hyperv-windows` و`hyperv-linux`: إنشاء/استعادة checkpoint وتشغيل أوامر عبر قناة إدارة موثقة.
- `wsl2-linux`: بديل محلي للقبول الدوري، وليس دليل clean-host نهائيًا.
- `external`: إدخال manifest ودليل صادرين من جهاز قبول خارجي دون ادعاء تشغيله محليًا.

كل provider يطبق واجهة موحدة: `prepare`, `install`, `start`, `exec`, `collect`, `reset`, `destroy`. عمليات reset/destroy تتطلب هدفًا صريحًا متحققًا منه ولا تعمل على host أو مجلد واسع.

### سائقا الأدوار

Playwright يقود UI بجلسات مستقلة للأدوار. وتستخدم طلبات API فقط لإعداد fixtures أو التحقق الخلفي عندما يكون ذلك موثقًا؛ لا تُستبدل الخطوة التفاعلية الأساسية بطلب API لأن الغرض قبول رحلة المستخدم.

### البيانات والأدلة

يولد benchmark generator البيانات ببذرة ثابتة، ويضع الملفات في مساحة مؤقتة محددة. تُحفظ الأدلة خارج شجرة المصدر أو في مجلد artifacts متجاهل، ويمنع فحص النظافة بقاء `test-results` أو `playwright-report` داخل المستودع.

## سياسات التنفيذ

- السيناريوهات موسومة `smoke`, `daily`, `nightly`, `rc`, `ga`, و`external`.
- القدرات غير المتوفرة تُسجل `blocked-capability` مع السبب، لا `passed` ولا `failed`.
- السيناريو الذي يفشل يعيد المحاولة مرة واحدة فقط إذا صُنّف flake آليًا؛ النتيجتان تحفظان.
- لا تُغلق مهام clean-host بنتيجة Docker أو WSL2.
- لا يُشغّل اختبار حمل كثيف على جهاز التطوير دون عقد موارد وحدود قرص وزمن.
- يوقف المنسق كل الخدمات والـVMs التي بدأها في `finally`، مع خيار صريح لإبقاء البيئة للتشخيص.

## بوابات الإصدار

- **دفعة يومية:** smoke Docker + رحلة تسجيل الدخول والبحث وفتح السجل والصحة.
- **Nightly:** Docker كامل + VM متاح + أدوار متزامنة + حمل متوسط.
- **RC:** المصفوفة الكاملة على Windows/Linux/Docker، online/offline، backup/restore، update/rollback، game-day، والأحمال المعتمدة.
- **GA:** إعادة RC من artifacts النهائية الموقعة على clean hosts، بلا تغييرات أو build جديد.

يفشل RC/GA عند وجود فشل وظيفي، تسرب صلاحيات، فساد أو فقد بيانات، تجاوز أداء غير مستثنى، دليل ناقص، أو سيناريو مانع لم يُنفذ. `blocked-capability` مقبول يوميًا لكنه مانع في RC/GA إذا كان السيناريو مطلوبًا للمنصة المعلنة.

## التوافق مع خطة V1 الحالية

هذه المصفوفة لا تستبدل المهام الموجودة، بل تجعل قبولها قابلًا للقياس:

- تربط fixtures والأدوار بـV1-303B وتغطية المسارات والوصولية بـV1-303C..E.
- تستخدم generator وbudgets في V1-307A..D.
- توسع game-day في V1-501 وتقدم دليل V1-503 وV1-505.
- تجمع أدلة clean-host اللازمة لـV1-210D وV1-211D وV1-502 وV1-604.
- تصبح بوابة دخل لـV1-601 Go/No-Go ولا تسمح بإغلاقها بأدلة محلية جزئية.

## معايير اكتمال البرنامج

يُعد برنامج القبول جاهزًا عندما:

1. توجد قائمة سيناريوهات versioned تغطي الحزم الست دون فجوات مانعة.
2. يعمل منسق واحد على Docker ويدعم provider واحدًا على الأقل لـWindows وواحدًا لـLinux.
3. تنتج كل رحلة تقريرًا موحدًا قابلًا للقراءة آليًا والبشرية مع أدلة منزوعة الأسرار.
4. تنفذ جلسات Playwright مستقلة للأدوار ولا تعتمد على token مدير موحد.
5. يمكن إعادة run فاشل ببذرته وmanifest نفسه.
6. يثبت RC وGA النجاح من artifacts الموقعة على clean hosts وفق المصفوفة المعلنة.
